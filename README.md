# Sonara

Sonara is a hackathon prototype for assistive navigation that turns a live camera feed into spatial audio and haptic cues. It uses monocular depth estimation plus object detection to help a user understand where nearby obstacles are, then plays directional sound on the phone and triggers vibration when something is too close.

Built by team `Zombie` for a hackathon sprint, Sonara is intentionally optimized for a live demo: fast setup, clear feedback, and a pipeline that still works even when object detection misses something.

## What It Does

- Estimates scene depth from a single RGB camera feed using MiDaS.
- Detects known objects with YOLOv8n for extra semantic context.
- Splits the view into a `3 x 2` grid: left, center, right × upper, lower.
- Computes proximity for all 6 zones, even if no object label is detected.
- Converts the nearest active zones into spatially-positioned audio cues in the browser.
- Triggers a danger click and phone vibration for very close obstacles.
- Supports both an external IP Webcam feed and the phone's own browser camera.
- Optionally seeds oscillator micro-variation with true quantum random numbers to reduce listener fatigue.
- Streams an annotated output feed for easier debugging and judging.

## Why The Current Version Is Better

The older prototype focused on only a few detected objects. The current build is more reliable for obstacle awareness because depth drives the experience first, and detection labels are layered on top when available.

That means:

- Sonara still reacts when YOLO sees nothing.
- The user gets consistent left/center/right and upper/lower guidance.
- Nearby walls, furniture, and unlabelled obstacles can still trigger warnings.
- The demo is easier to explain because every part of the frame maps to a zone.

## How It Works

```text
Phone camera or IP Webcam
        |
        v
Laptop processing app (Streamlit)
  - MiDaS depth estimation
  - YOLOv8n object detection
  - 6-zone scene analysis
        |
        +--> annotated MJPEG video stream
        |
        +--> static/scene.json with live obstacle state
                     |
                     v
Phone browser audio component
  - Stereo spatial audio
  - Danger beep
  - Vibration feedback
```

## Core Pipeline

1. A frame comes from either an MJPEG stream or the phone browser camera.
2. The frame is resized and passed through MiDaS to estimate relative depth.
3. YOLOv8n detects objects and provides labels where possible.
4. `scene.py` divides the depth map into 6 regions and scores each by proximity.
5. Each region becomes a structured output with: `zone`, `level`, `proximity`, `distance_m`, `label`, `bbox`, `danger`.
6. The app writes this state atomically to `static/scene.json`.
7. The browser polls that file every 100ms and sonifies the nearest active zones.

## Key Features

- `Depth-first obstacle awareness`
  Sonara does not depend on object classes. If the model sees something near but cannot name it, the system still warns the user. Walls, glass doors, random furniture — all caught.

- `6-zone spatial mapping`
  The scene is divided into upper/lower and left/center/right regions. Audio cues are more intuitive during movement because the sound position matches where the obstacle actually is.

- `Stereo spatial audio`
  The browser uses Web Audio StereoPannerNode to position sounds left or right. Nearer objects produce higher-pitched tones. Left/right placement matches the obstacle direction in the frame.

- `Danger haptics`
  High-risk zones (proximity > 0.75) trigger a sharper 1800Hz alert tone and phone vibration with a 600ms cooldown to prevent spam.

- `Quantum-seeded audio texture`
  When enabled, each oscillator's frequency is micro-varied by ±15Hz using true random numbers fetched from the ANU Quantum Random Numbers API — values generated by measuring quantum vacuum fluctuations. This prevents listener fatigue from flat synthetic tones during extended use. Toggled from the sidebar, off by default, falls back silently if the API is unreachable.

- `Two camera modes`
  Use an IP camera app for easy local-network streaming, or use the phone's own camera directly from the browser.

- `Judge-friendly visualization`
  The laptop displays the annotated stream with a depth heatmap overlay, zone boxes, object labels, and estimated distances while the phone handles audio and haptics independently.

- `Apple Silicon optimized`
  The depth model runs on MPS when available, falling back to CUDA then CPU. On an M-series MacBook, this gives a meaningful FPS improvement over pure CPU inference.

## Tech Stack

|-----------|---------|

## Setup

### 1. Create a virtual environment

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 2. Start the app

```bash
streamlit run app.py --server.address=0.0.0.0
```

Streamlit will print a local URL and a network URL. Open the **network URL** on your phone — this is required for audio and vibration to work on the actual device.

### 3. Choose a video source

#### Option A: IP Webcam (recommended)

- Install IP Webcam (Android) or EpocCam (iPhone) on the phone.
- Start the server in the app and note the stream URL, usually ending in `/video`.
- Paste that URL into Sonara's sidebar.
- Toggle on `Start Camera Processing`.

#### Option B: This device camera

- Open Sonara on the phone browser directly.
- Switch the source to `This device camera`.
- Toggle on `Start Camera Processing` and allow camera access when prompted.

> Note: many mobile browsers require HTTPS or localhost for direct camera access. If plain `http://` does not work, use the IP Webcam route instead.

## Demo Flow

For a live hackathon demo:

1. Run the app on the laptop: `streamlit run app.py --server.address=0.0.0.0`
2. Open the **network URL** on the phone browser.
3. Tap `Enable Audio + Haptics` on the phone.
4. Start camera processing from the sidebar.
5. Move the phone toward obstacles — the judges should hear spatial cue changes clearly through earphones.
6. Keep the laptop screen visible so judges can see the annotated zones, distances, and depth heatmap in real time.
7. Before the judging session, enable `⚛️ Quantum audio texture` in the sidebar for the quantum RNG integration demo point.

## Current Output Format

Each zone in the live scene state:

```json
{
  "zone": "left",
  "level": "upper",
  "proximity": 0.84,
  "distance_m": 0.78,
  "label": null,
  "bbox": [0, 0, 128, 144],
  "danger": true
}
```

This is the shared contract between the Python inference pipeline and the browser audio system.

## Limitations

- Depth is monocular and approximate, not true metric depth.
- Direct phone-camera mode may fail on non-HTTPS mobile sessions.
- Vibration support varies by browser and device.
- Performance on CPU-only machines is lower than on Apple Silicon or CUDA systems.
- This is a hackathon build — robustness and accessibility testing are still limited.
